redis高阶特性

发表于 2018-09-20 | 分类于 redis | 阅读次数:

介绍

redis的高阶特性：分布式lock，delay queue, bitmap, hyperloglog, rebloom, redis-cell, geohash

github代码：https://github.com/chenlisong/projects/tree/master/redis4util

版本：>= 4.0

分布式lock

  • 加过期时间，防止进程挂掉，导致锁一直被占用，无法释放

  • setnx ex需要原子操作执行。新版本的set方法，老版本的SetParam，lua都可以实现

  • value值随机，unlock时对比。防止出现其他进程unlock另外一个进程的lock，比如执行时间 > expireTime。

  • 关于预计ex时间不准的问题，因为ex解决的是锁一直被占用的问题，可以考虑使用daemon进程心跳ex

delay queue

  • redis的list可以当成异步消息队列使用，lpop，rpush

  • blpop，阻塞读，在没数据的时候休眠，有数据时立刻醒过来，降低延迟。需要捕获异常，重试

使用zset结构实现一个延迟队列

bitmap

每一位值只有0或者1，例如setbit field index 1。

  • bitfield w get u4 0 # 从第一个位开始取 4 个位，结果是无符号数 (u)

  • bitfield w get u3 2 # 从第三个位开始取 3 个位，结果是无符号数 (u)

  • bitfield w get i4 0 # 从第一个位开始取 4 个位，结果是有符号数 (i)

  • bitfield w get i3 2 # 从第三个位开始取 3 个位，结果是有符号数 (i)

HyperLogLog

  • add元素，count数量，非精确，适合大量统计，允许丢失精度。

  • 存储空间少，需要12k。16384桶 * 6bits /8 = 12k

rebloom

HyperLogLog能解决count的问题，但是不能解决contains的问题。rebloom就是解决contains的问题。

  • 添加key时，通过m次f(n)计算出几个数组index，写上1

  • 判断是否存在时，执行同样m次f(n)，如果都是1，代表存在

  • 有误杀，但是不会放过曾经写入的

  • 误差率可以通过内存的增大，配置修改进行调整。布隆计算器

geohash

  • 将世界地图不断的缩小2倍分割，直到分割的面积到足够小

  • 计算两点距离，计算附近的点，支持排序

geoadd geopos geolist georadiusbymember

redis-cell

redis4支持module插件，redis-cell是以第三方插件，安装后更新redis配置，支持漏斗限流。
