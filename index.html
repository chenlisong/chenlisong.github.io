redis常用的数据结构之ziplist

发表于 2018-09-19 | 分类于 redis | 阅读次数:

介绍

redis常用的数据结构ziplist，及其底层数据结构实现。

涉及的内容：ziplist, intset

版本：>= 4.0

ziplist

Redis 为了节约内存空间使用，zset 和 hash 容器对象在元素个数较少的时候，采用压缩列表 (ziplist) 进行存储。压缩列表是一块连续的内存空间，元素之间紧挨着存储，没有任何冗余空隙。

ziplist的数据结构

因为ziplist是紧凑存储，所以插入元素时需要remalloc

keys

  • 压缩列表为了支持双向遍历，所以才会有 ztail_offset 这个字段，用来快速定位到最后一个元素，然后倒着遍历。

  • 如果entry.content小于254字节，prevlen是1个字节，内容大于254则prevlen是5个字节。所以list中内容变化可能导致级联更新的问题。

intset

数据结构如下：

当 set 集合容纳的元素都是整数并且元素个数较小时，Redis 会使用 intset 来存储结合元素。intset 是紧凑的数组结构，同时支持 16 位、32 位和 64 位整数。

  • public synchronized void lockRead()

  • public synchronized void unlockRead()

  • public synchronized void lockWrite()

  • public synchronized void unLockWrite()

且都是synchronized的方法，因为其在实例级别并发下，需要保持先后顺序。

  • lockRead时，不能有写，写等待请求。

  • unlockRead时，无要求

  • lockWrite时，不能有读，写请求

Java堆外内存，占用系统的os内存排查

发表于 2018-09-10 | 分类于 java | 阅读次数:

通过对堆外内存的分析，进程内本地方法的调用，追踪。找到Java进程占用大量内存的原因并解决。

使用的相关工具

  • jvisualVM：查看heap内存变化，堆外内存变化

  • pmap pid: 查看进程占用的内存

  • jemalloc: 查看进程对native方法的调用，以及对调用次数的统计，占比。

  • btrace: 拦截指定的本地方法，追踪调用链。

问题描述

  1. 进程启动时正常，进程的内存占用在600m左右；

  2. 持续的烤机压测下，进程占用的内存持续线性上涨，内存到12g后，查看dmesg和/var/log/messages，被系统强杀；

  3. cpu正常，负载正常 56%；

  4. heap内存2g，gc正常，堆外内存17m，均正常；

  5. top下res内存持续上涨，涨至12g，被系统自动杀掉。

推测：由Java进程发起调用linux native方法，导致进程额持有的os内存占用持续飙涨。

Java gc总结

发表于 2018-07-12 | 分类于 java | 阅读次数:

介绍

长夜漫漫，无心睡眠，起来总结一下Java的gc，cms，g1, zgc。以供回顾。

CMS gc

常见垃圾回收器

  • 串行 Serial 单线程的一个回收器，简单，易实现，效率高

  • 并行 ParNew Serial的多线程版，可以充分的利用cpu的资源，减少回收的时间

  • 吞吐量优先 Parallel Scavenge，侧重于吞吐量的控制

  • 并发标记清除 CMS concurrent mark sweep 一种以获取最短回收停顿时间为目标的回收器，该回收器基于标记-清除算法

cms的执行过程

  • init-mark初始标记 stw，该阶段进行可达性分析，标记gc root能直接关联到的对象，很快

  • concurrent-mark 并发标记，由前段标记过的绿色对象出发，所有可达到的对象都在本阶段标记

  • remakr重标记，stw，暂停所有用户线程，重新扫描用户堆对象，进行可达性分析，标记活着的对象

    因为阶段2，可能有用户线程重新让一个对象变得可达。注意，当前阶段以新生代对象为根来判断对象是否存活

  • 并发清理，进行并发的垃圾清理
